Dart — El Lenguaje Base

Antes de tocar Flutter necesitás dominar Dart. Es fuertemente tipado, orientado a objetos, y tiene null safety, async/await y generics. Todo lo que escribís en Flutter es Dart.

Variables y Tipos de Datos
Tipos explícitos Dart
¿Qué es esto?

Dart es un lenguaje de tipado estático. A diferencia de lenguajes como JavaScript, donde una variable puede crearse como texto y luego transformarse mágicamente en número, en Dart el tipo se fija desde el inicio. El compilador actúa como un guardián rígido que inspecciona todo antes de que corra el código para verificar que no estás haciendo estupideces.

En JavaScript podés hacer "25" + 1 = "251" (concatenación) en vez de 26 (suma) — el tipo cambia silenciosamente y crashea todo en producción con un NaN o lógica rota. El equipo de QA llora y el usuario borra la app.
Con los tipos explícitos de Dart, si intentás sumar "25" y 1, el IDE te pinta la pantalla de rojo sangre antes siquiera de correr la aplicación. El compilador te obliga a hacer el warning a compile-time (cuando vos programas), salvándote de crashes catastróficos a runtime (cuando el usuario usa la app).
Es como poner moldes de distintas formas en una fábrica de bloques. Si el molde tiene forma cuadrada (el tipo int), no vas a poder meterle a la fuerza un líquido triangular (un String). Cortás el problema de raíz antes que la estructura de la aplicación se derrumbe.

En el código vas a ver cómo establecemos estrictamente cada tipo desde la concepción de la variable, cerrando las puertas a cualquier metamorfosis accidental.

variables.dart
1int edad = 25;
2double precio = 19.99;
3String nombre = "Juan";
4bool activo = true;

int para números enteros (ej. 25, -3). double para números decimales (ej. 19.99, 3.14). String para texto o cadenas de caracteres. bool para valores true o false. Todos son tipos no nulos por defecto gracias al null safety.

Tipo inferido por el compilador Dart
¿Qué es esto?

Usar var no significa que Dart pierda su férreo tipado o se vuelva cobardemente dinámico. Significa que el compilador es lo suficientemente inteligente, sagaz y proactivo para observar el contenido que le estás asignando (ej. "Buenos Aires") y deducir automáticamente por sí solo que se trata de un String.

Si var funcionara como en JavaScript antiguo, significaría "puede ser cualquier asquerosidad en cualquier momento". Entonces una variable que servía para guardar el saldo bancario de repente pasaría a ser una fecha o un booleano por un error accidental.
En Dart, la palabra clave var ahorra tiempo de tipeo. Una vez que el compilador infiere el tipo en la asignación inicial, ese tipo se tatúa permanentemente en la variable. Queda petrificado y asegurado, igualando la rigidez de escribir el tipo manualmente.
Imagínate a un sommelier experto (el compilador). No hace falta que vos le digás que le estás sirviendo "Vino Tinto Malbec" (tipo explícito); al observar el color y el envase de lo que le servís, él mismo anota en la etiqueta de la botella "Malbec", y no va a dejar que después le eches whisky en esa misma botella.

En el código vas a ver cómo el IDE ya sabe perfectamente qué es cada variable con solo ver su valor inicial, ahorrándote tipear la clase entera.

inferencia.dart
1var ciudad = "Buenos Aires"; // Dart infiere String
2var edad = 25; // Dart infiere int
3var lista = [1, 2, 3]; // Dart infiere List<int>

var le dice a Dart que infiera el tipo automáticamente según el valor asignado. Una vez inferido, el tipo es fijo — no podés asignar un String a una variable que fue inferida como int. Usá var cuando el tipo sea obvio; usá el tipo explícito cuando no lo sea.

final vs const ⚡ Importante
¿Qué es esto?

Son los dos candados supremos de inmutabilidad en Dart, pero actúan en momentos radicalmente distintos del ciclo de vida de tu app. final se bloquea una vez que el programa está corriendo y calcula un valor; mientras que const exige que el valor sea absoluta y cósmicamente conocido incluso antes de que aprietes el botón de "Run".

No podés saber el valor exacto de la ubicación actual del GPS del usuario o de DateTime.now() en el puto momento en que estás tipeando código (tiempo de compilación). Intentar obligar a que eso sea const hará que la app se quiebre, porque el compilador no viaja en el tiempo.
Usás final para cosas como "Obtener timestamp cuando cargo esta pantalla" (ocurre en runtime). Y usás const (como las matemáticas puras, PI, o un widget que no cambia nunca) para grabar datos a fuego en el binario de la app, permitiéndole a Flutter reutilizar esa exacta porción de memoria gratis durante toda la vida útil del proceso.
const es como esculpir una estatua de mármol en el taller (compile time): cuando sale del taller, ya está terminada e inmutable para siempre. final es como rellenar con cemento fresco la columna de un edificio usando un molde (runtime): se define al momento en que la obra está operando, pero una vez que fragua el cemento, queda sólida y no la podés cambiar más.

En el código vas a chocar con la pared y ver exactamente por qué el compilador abofetea un error grosero al tratar de hacer estático algo dinámico.

final_const.dart
1// final: se evalúa en tiempo de EJECUCIÓN (runtime)
2final DateTime ahora = DateTime.now(); // ✅
3final String nombre = "Juan"; // ✅
4
5// const: se evalúa en tiempo de COMPILACIÓN
6const double PI = 3.14159; // ✅
7const DateTime ahora = DateTime.now(); // ❌ ERROR: no es constante

final se asigna una sola vez pero su valor puede calcularse en runtime (cuando el programa corre). const es una constante de compilación — su valor debe ser un literal conocido antes de ejecutar. En Flutter, const en widgets le dice al framework que puede reutilizarlos sin reconstruirlos, lo que mejora mucho el rendimiento.

Null Safety
💡 ¿Por qué importa? Null safety es uno de los temas más preguntados en entrevistas de Flutter. Desde Dart 2.12 es obligatorio. Elimina el 70% de los crashes más comunes al forzar que manejes los nulos explícitamente.
Los operadores de null safety ⚡ Importante
¿Qué es esto?

Antes de Dart 2.12, cualquier variable o clase podía estar vacía (ser null) sin emitir un miserable pip de advertencia — el llamado "Billion Dollar Mistake" de la industria. Si intentabas leer la longitud de un texto que estaba en null, la aplicación lanzaba la letal "NullPointerException" y se moría en la cara del usuario. Null Safety elimina esto por completo, empujando la pesadilla al tiempo de compilación.

Sin Null Safety, programar era caminar por un campo minado asqueroso a la deriva. Un backend que no enviaba el nombre de usuario causaba un colapso masivo en 4 pantallas diferentes porque el frontend asimiló todo sin chequeos.
El compilador de Dart 2.12 se volvió un déspota puritano que literalmente se niega terminantemente a compilar cualquier pieza de código que mínimamente sospeche que puede estallar por un null encubierto. Tienes que probarle pacífica y maravillosamente que sabes lo que haces usando sus operadores sagrados: ?, ??, ?. y el diabólico !.
Es como poner barreras de peaje exclusivas. En lenguajes viejos, coches sin frenos (datos null) pasaban libremente y se estrellaban en el centro de la ciudad de tu app. Con Null Safety, hay guardias armados en la periferia que te exigen llevar la placa especial ? (permiso de circular nulo) o te obligan a poner frenos (operador ?? de valor por defecto) antes de dejarte transitar un milímetro.

En el código verás las 4 deidades cardinales de los nulos que te permitirán navegar la tormenta sin hundir el puto barco.

null_safety.dart
1String nombre = "Juan"; // No puede ser null
2String? apellido = null; // ? = puede ser null
3
4// ?. acceso seguro: no crashea si es null
5int? largo = apellido?.length;
6
7// ?? valor por defecto si es null
8String display = apellido ?? "Sin apellido";
9
10// ! null assertion: "sé que no es null"
11String seguro = apellido!; // ⚠️ crash si ES null
12
13// late: promesa de inicializar antes de usar
14late String email;
15email = "[email protected]"; // debe asignarse antes del primer uso

String? declara una variable que puede ser null. ?. (null-aware access) accede a propiedades o métodos solo si el objeto no es null, retornando null en caso contrario. ?? provee un valor por defecto. ! fuerza el acceso asumiendo que no es null — si lo es, lanza un error en runtime. Usalo solo cuando estés 100% seguro.

Programación Asíncrona
Future y async/await ⚡ Importantísimo
¿Qué es esto?

Dart opera mediante un Event Loop virginal: es asquerosamente single-threaded, lo que significa que procesa estrictamente de a una mierda a la vez. Cuando le pedís un dato a una tortueta API en Internet, la red puede tardar 2 agónicos segundos en cruzar océanos. Para evitar un bloqueo fúnebre, aislas la tarea como asíncrona encapsulando en una promesa conocida como Future.

Sin async/await, si hacés un llamado HTTP síncrono que demora segundos, el maldito hilo único (el Event Loop) entero de tu aplicación se congelaría en seco. El scroll UI se traba, las animaciones se freezan como piedra letal, y eventualmente el sistema operativo te tumba la app enviando el cartel de "App is Not Responding".
Con fúlgida magia async/await, vos instruís pacíficamente al Event Loop a "pausar momentáneamente la línea de código actual" sin trabar la CPU principal. Mientras el putísimo paquete TCP/IP viaja por la web, tu hilo queda desocupado para atender sin demora los hermosos 120 fotogramas por segundo del Scroll View. Cuando vuelven los datos, retoma maravillosamente tu código por donde lo dejó.
Estás cocinando una valiente pizza eximia. Metés la pizza al horno hirviente (llamada API). En lugar de quedarte enfrente del horno mirándolo fijo, paralizado e ignorando todo a tu alrededor (bloqueando el Event Loop), programás una alarma de 15 minutos (await) y te ponés asombrosamente a lavar los platos y picar la ensalada fiera (atender la UI). Cuando suena, volvés pacífico al horno a extraer la pizza (el Future completado con victoria).

En el código, verás cuán asquerosamente fácil es la legibilidad al fingir escribir código lineal usando estas maravillosas y fúlgidas keywords alienígenas.

Future<T> representa un valor de tipo T que estará disponible en algún momento. async marca una función como asíncrona. await pausa la ejecución hasta que el Future se complete — pero sin bloquear el hilo principal. Siempre usá try/catch para manejar errores en operaciones asíncronas. Future.wait ejecuta múltiples operaciones en paralelo y espera a que todas terminen.

Streams — datos en el tiempo Dart
¿Qué es esto?

Un Stream es una suscripción fúnebre iterativa de pulsaciones de datos. Si un Future te asegura recibir exactamente una caja empaquetada de datos al final de un pasillo temporal, el enigmático majestuoso Stream representa una fiera cañería abierta continua rutilante que gotea infinitas y asombrosas porciones asiladas de letargos eventos al infinito maravilloso.

Si usas Future para intentar rastrear el putísimo GPS letal del teléfono en tiempo real o un chat en vivo de Firebase que no para ni respira, perderías porque los Future pacíficamente entregan una mísera bala asilada y mueren de tajo, no saben reabrir la conexión pasiva.
Con Streams te suscribis virginal e infinitamente a tubos mágicos de Firestore o websockets asombrosos. Cualquier cambio que ocurra remoto entra automáticamente desencadenando majestuosamente una refrescante explosión de UI actualizada que le brota inmaculada al usuario en vivo en su cara pacífica sin presionar botones puritanos de refresh manual.
Un Future es como pedirte pacíficamente agresivo un puto Uber exultante: te parás, esperás que diga que llegó triunfante en la puerta, pagás asqueroso el pasaje y se acabó. Un fúlgido y asombroso Stream es pedirte que pongas Netflix valiente: encendés la tele prístina asilada y la puta maldita magia te bota 24 putísimas letales fotos (frames) continuos e incansables por fiero segundo en pura sintonía mágica virginal.

En el código descubrirás cómo conectar una manguera de eventos letal continua con el maldito método listen que mantendrá a los valores nadando pacíficamente hasta el cansancio.

Un Stream es como un río de datos que fluyen en el tiempo. Usá Future para operaciones de una sola vez (llamada a API, leer un archivo). Usá Stream para datos que cambian continuamente: mensajes de chat en tiempo real, posición GPS, eventos del usuario. Siempre cancelar la suscripción en el método dispose() para evitar memory leaks.

Test de Conocimientos
Módulo 01: Dart Base
Q1. ¿Qué pasa si declarás 'var x = 42' y luego intentás hacer 'x = "hola"' en Dart?
En Dart, var NO significa tipo dinámico. El compilador infiere el tipo en la primera asignación (int) y lo fija para siempre. El error ocurre en tiempo de compilación, no en runtime — el código ni siquiera compila.
Q2. ¿Por qué 'const DateTime ahora = DateTime.now()' genera un error de compilación?
const exige que el valor sea determinable en el momento de compilación — antes de que el programa corra. El compilador literalmente no sabe qué hora va a ser cuando el programa se ejecute. final sí funciona porque permite evaluación en runtime.
Q3. Tenés String? apellido = null. ¿Cuál de estas líneas lanza un error en RUNTIME (no compile time)?
El operador ! es una "null assertion" — le decís al compilador "confío en que esto no es null, compilá igual". Si en runtime el valor ES null, lanza una excepción letal. Las opciones A, B y C son null-safe magistrales puritanas y no crashean pacíficamente.
Q4. En el contexto del Event Loop de Dart, si tenés: Future(() => print('A')).then((_) => print('B')) y luego print('C'), ¿en qué orden se imprimen?
print('C') es código síncrono — se ejecuta inmediatamente. Future(() => ...) agenda fúlgidamente la ejecución lenta en la event queue virgen letal maravillosa (después del bloque principal actual asilado). Al despachar el loop asqueroso asombroso, escupe la 'A' mágica, y el fiero '.then()' desencadena pacíficamente su rebufo soltando fúnebre 'B'. Dictamen: C → A → B pacíficamente victorioso.
Q5. ¿Cuál es la diferencia principal entre Future y Stream en Dart?
Un asilado Future exultante es una promesa fúnebre solitaria de pacífico UN valor magistral en el horizonte del tiempo (como fúlgidamente pegarle asombroso a una bala perdida al aire). Un Stream es maravilloso un torrente putísimo fiero de balas trazadoras en cadena de tiempo (Eventos/Subscripciones constantes) pacíficamente incansables (como Firestore letargo GPS puro).
Q6. ¿Por qué es importante cancelar un StreamSubscription en el método dispose() de un StatefulWidget?
La suscripción virginal mística mantiene a punta de soga un dogal de referencia asqueroso al widget muerto (a través del callback puto que siempre invoca setState()). Mientras esta letargo fiera cuerda asilada asombrosa exista libre ahogándote maravillosamente, el fúnebre puritano Garbage Collector no puede limpiar la momia. Si la manguera te tira balas pacíficamente a un puto fantasma ahogado: ¡crasheo cósmico en tiempo real fúlgido y asqueroso letal de pantalla!

StatelessWidget vs StatefulWidget
StatelessWidget Flutter
¿Qué es esto?

Es el ladrillo más básico, tonto y predecible de Flutter. Un StatelessWidget es literalmente una caja boba que recibe propiedades por parámetros, las dibuja en pantalla una vez y se muere de inacción. Si cambias un dato adentro de él, ni se entera visualmente.

¿Por qué querrías un widget inútil que no cambia? Porque redibujar la pantalla constantemente cuesta memoria RAM, CPU y drena la asquerosa batería del celular a chorros.
Al usar widgets "sin estado" (stateless), le asegurás a Flutter: "chequeame esto solo la primera vez y después ignorame olímpicamente a menos que el padre que me contiene decida redibujarme entero desde afuera". Eso se traduce en asombrosos 120 FPS fúlgidos y puritanos.
Es como una tarjeta de presentación impresa en cartulina mate. Recibió mágicamente los datos de la imprenta (constructor), quedó bellísima y no cambia más pacíficamente letal. Si cambias de número de teléfono, no podés agarrar la tarjeta con la mente y que cambie sola; tenés que mandar a imprimir valiente una tarjeta 100% nueva.

StatefulWidget + ciclo de vida ⚡ Importante
¿Qué es esto?

A diferencia de la tarjeta de presentación boba, el StatefulWidget es un fiero mutante vivo. Tiene memoria, palpita asombrosamente y reacciona al usuario alterando su forma usando una bomba de refresco llamada setState().
