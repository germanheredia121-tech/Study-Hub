Módulo 01

Complejidad Algorítmica — Big O y Space Complexity

El lenguaje universal de los ingenieros para medir y comunicar qué tan eficiente es un algoritmo en términos de Tiempo y Memoria cuando la cantidad de datos (input) crece hacia el infinito. Sin dominar esto, jamás sabrás si una solución es óptima o si pasará la entrevista.

Time Complexity (Tiempo de Ejecución)

Mide cómo aumenta la cantidad de operaciones que hace tu programa a medida que el input (N) se vuelve gigante. En Big O, siempre nos preparamos para el peor escenario (el peor caso) y utilizamos una heurística vital: las constantes se ignoran por completo.

Las 6 Complejidades Clásicas vital
EXPLICACIÓN TEÓRICA EXHAUSTIVA

¿Por qué ignorar constantes? Si tienes un algoritmo que hace 2 * N operaciones (o sea, O(2n)), decimos que es estrictamente O(N). Porque si N es un trillón, ese "x2" no cambia la forma geométrica del crecimiento en la gráfica matemática; sigue siendo una línea recta (lineal). La regla de oro en Big O es que las constantes no importan: O(2n) -> O(n) y O(n + 500) -> O(n).

Las Reglas del Juego Lógico
💡 Analogía real para entender Constant vs Log vs Lineal:
Buscar la palabra "Cebra" en un diccionario impreso de papel:
- O(n): Leer el diccionario página por página minuciosamente desde la A. Te va a tomar años de encierro si el libro tiene millones de páginas.
- O(log n): Abrirlo por la mitad exacto. Si ves la letra "M", sabes que Cebra está en el cerro de la izquierda. Descartas matemáticamente la mitad derecha entera y repites abriendo la mitad izquierda. Solo necesitas ~20 iteraciones para encontrarla en un libro de 1 millón de páginas (Como hace Binary Search).
- O(1): Tienes el número de página anotado en la palma de tu mano izquierda y vas directo. Tardas 1 segundo milimétricamente no importa qué tan grueso sea el diccionario.

Space Complexity (Complejidad de Espacio)
🤔 ¿Por qué importa el espacio y cómo se calcula?
Time Complexity mide cuán veloz eres, pero Space Complexity evalúa cuánta memoria RAM adicional (auxiliar) necesitas para operar. Atención vital: La matriz u objeto que te proveen originalmente como Input no cuenta como espacio adicional. Cuentan únicamente los constructos extra, diccionarios extra, matrices y recursos matemáticos que debas instanciar para resolver tu dilema.
Memoria Auxiliar vs Creación de Copias memory
¿QUÉ OCUPA ESTE ESPACIO EN LA RAM?

Hay exactamente dos grandes vampiros de recursos de memoria en cualquier paradigma de algoritmos:

  • Creación de copias y nuevas estructuras: Si el Input es un grandioso Array de tamaño N, y tú para solventar un problema decides internamente programar algo que crea una copia de ese array, habrás condenado a tu algoritmo silenciosamente a consumir un Espacio Auxiliar de O(n). Un mapa Hash/diccionario con N llaves también ocupa mágicamente O(n) espacio.
  • El Call Stack en Recursiones (La Pila Interna del Sistema del CPU): El SO de un computador preserva el registro de cada llamada a una función dentro del Call Stack antes de regresar arriba. Si desencadenas una Recursión profunda sin salir que se profundiza con tamaño N, consumiste un Espacio de Sistema real en el Stack equivalente a O(n), aunque tu sintaxis ni si quiera tenga variables.

Módulo 02

Arrays & Strings

La base de casi todo. El 60% de los problemas de LeetCode involucran arrays. Dominá estos patrones primero.

Patrón: Two Pointers
💡 Cuándo usar Two Pointers Cuando el array está ordenado y necesitás encontrar pares o tripletas que cumplan una condición. Convierte problemas O(n²) en O(n). Señal en el enunciado: "sorted array", "find pair that sums to X", "remove duplicates".
Two Sum en array ordenado O(n) · O(1)
two_sum_sorted.py Python
1# Input: [2,7,11,15], target=9 → [0,1]
2def two_sum_sorted(nums, target):
3 left, right = 0, len(nums) - 1
4 while left < right:
5 current = nums[left] + nums[right]
6 if current == target:
7 return [left, right]
8 elif current < target:
9 left += 1 # necesitamos más: mover izquierda
10 else:
11 right -= 1 # necesitamos menos: mover derecha
12 return []

Dos punteros que se mueven desde los extremos hacia el centro. Si la suma es menor que el target necesitamos un número mayor → movemos el puntero izquierdo. Si es mayor → movemos el derecho. Solo funciona con arrays ordenados. O(n) tiempo O(1) espacio

3Sum — tripletas que suman cero O(n²) · O(1)
EXPLICACIÓN TEÓRICA

¿Qué es esto? Es una extensión de Two Pointers. Fijamos un elemento con un loop (ese es el primero de la tripleta) y luego usamos Two Pointers en el resto del array para encontrar el par que completa la suma en cero.

Buscando tres personas que sumen un peso exacto: elegís a una fija, y luego ponés a dos ayudantes a buscar las otras dos en el resto del grupo ordenado.
El sorting previo es obligatorio. Sin skipear repetidos retornaríamos [-1,-1,2] dos veces si hay dos -1 en el array original. Esto soluciona O(n³) llevándolo a O(n²).

Observarás el loop principal y cómo usamos 'continue' obligatoriamente para saltarnos duplicados.

three_sum.py Python
1def three_sum(nums):
2 nums.sort() # CRUCIAL: ordenar primero
3 result = []
4 for i in range(len(nums) - 2):
5 if i > 0 and nums[i] == nums[i-1]: continue # skip dup
6 left, right = i + 1, len(nums) - 1
7 while left < right:
8 s = nums[i] + nums[left] + nums[right]
9 if s == 0:
10 result.append([nums[i], nums[left], nums[right]])
11 while left < right and nums[left] == nums[left+1]: left += 1
12 while left < right and nums[right] == nums[right-1]: right -= 1
13 left += 1; right -= 1
14 elif s < 0: left += 1
15 else: right -= 1
16 return result

Fijamos un elemento con el loop exterior, y usamos Two Pointers para el par restante. Lo crítico es skipear duplicados en las tres posiciones para no retornar tripletas repetidas. La poda del continue en línea 5 evita procesar el mismo número fijo más de una vez.

Patrón: Sliding Window
Minimum Window Substring Hard · O(s+t)
min_window.py Python
1# Input: s="ADOBECODEBANC", t="ABC" → "BANC"
2from collections import Counter
3def min_window(s, t):
4 need = Counter(t) # cuántos chars necesitamos
5 have, formed = {}, 0
6 required = len(need) # chars distintos a satisfacer
7 left = 0
8 result = float("inf"), -1, -1
9 for right in range(len(s)):
10 char = s[right]
11 have[char] = have.get(char, 0) + 1
12 if char in need and have[char] == need[char]:
13 formed += 1
14 while formed == required: # ventana válida: tratar de reducir
15 if right - left + 1 < result[0]:
16 result = (right - left + 1, left, right)
17 have[s[left]] -= 1
18 if s[left] in need and have[s[left]] < need[s[left]]:
19 formed -= 1
20 left += 1
21 return "" if result[0] == float("inf") else s[result[1]:result[2]+1]

Patrón de ventana deslizante variable. Expandimos con right hasta tener todos los chars necesarios. Una vez que la ventana es válida (formed == required), intentamos reducirla moviendo left hasta que deje de ser válida. Guardamos el mínimo en cada contracción. O(s+t)

Módulo 03

Linked Lists

Fundamentales en entrevistas. Te enseñan a pensar con punteros — habilidad que se transfiere a árboles y grafos. El truco del dummy node y fast/slow pointers son los patrones clave.

Patrón: Fast & Slow Pointers
Detectar ciclo — Floyd's Algorithm O(n) · O(1)
detect_cycle.py Python
1# Slow avanza 1, Fast avanza 2. Si hay ciclo, se encuentran.
2def has_cycle(head):
3 slow = fast = head
4 while fast and fast.next:
5 slow = slow.next
6 fast = fast.next.next
7 if slow == fast: return True
8 return False
9
10# Nodo medio: cuando fast llega al final, slow está en el medio
11def middle_node(head):
12 slow = fast = head
13 while fast and fast.next:
14 slow = slow.next
15 fast = fast.next.next
16 return slow
17
18# Reversar lista — el más importante de linked lists
19def reverse_list(head):
20 prev = None
21 curr = head
22 while curr:
23 next_node = curr.next # 1. guardar next
24 curr.next = prev # 2. apuntar hacia atrás
25 prev = curr # 3. avanzar prev
26 curr = next_node # 4. avanzar curr
27 return prev # nuevo head

Fast & Slow pointers es el patrón definitivo de linked lists. slow avanza 1 paso, fast avanza 2. Si hay un ciclo, eventualmente se encuentran — como dos corredores en una pista circular. Para reversar, siempre en el orden: guardar next → apuntar atrás → avanzar prev → avanzar curr. Cambiarlo de orden rompe la lista.

Merge K Sorted Lists — Hard con Heap Hard · O(N log k)
merge_k_lists.py Python
1import heapq
2def merge_k_lists(lists):
3 heap = []
4 # Meter el primer nodo de cada lista
5 for i, node in enumerate(lists):
6 if node:
7 heapq.heappush(heap, (node.val, i, node))
8 dummy = ListNode(0)
9 curr = dummy
10 while heap:
11 val, i, node = heapq.heappop(heap) # el mínimo
12 curr.next = node
13 curr = curr.next

15 heapq.heappush(heap, (node.next.val, i, node.next))

Usamos un min-heap de tamaño k (cantidad de listas). En cada paso, extraemos el nodo mínimo y agregamos el siguiente nodo de esa misma lista. El heap siempre tiene a lo sumo k elementos. O(N log k) donde N = total de nodos. Sin heap sería O(Nk). La tupla (val, i, node) incluye el índice de lista para desempatar cuando hay valores iguales.

Módulo 04

Stacks, Queues y Monotonic Stack

Stack (LIFO) y Queue (FIFO) son simples. El Monotonic Stack es el patrón avanzado más pedido en Hard — aparece en problemas de temperaturas, histogramas y agua atrapada.

Patrón: Monotonic Stack
💡 Cuándo usar Monotonic Stack Señales: "siguiente mayor/menor elemento", "días hasta temperatura más alta", "área máxima en histograma". El stack mantiene elementos en orden monotónico (creciente o decreciente). Cada elemento entra y sale del stack exactamente una vez → O(n).
Trapping Rain Water — 3 soluciones Hard · Icónico
EXPLICACIÓN TEÓRICA

¿Qué es esto? Tenemos un histograma y queremos saber cuánta agua quedaría atrapada si lloviera.

Como un balde: el volumen de agua en cada posición está rigurosamente limitado por el mínimo entre la pared más alta a su izquierda absoluta y la pared más alta a su derecha absoluta. Lo que importa es el lado más bajo.
Sin esto computarías y buscarías máximos a I y D perdiendo O(N²). Precomputar ambos lados gasta O(N) espacio. Two Pointers baja el espacio magistralmente a O(1).

Prestá atención a cómo comparamos max_left y max_right in-place letalmente desde ambos bordes.

trapping_rain.py Python
1# Input: [0,1,0,2,1,0,1,3,2,1,2,1] → 6
2
3# SOLUCIÓN 1: Precompute — O(n) time, O(n) space
4def trap_v1(height):
5 n = len(height)
6 left_max = [0] * n
7 right_max = [0] * n
8 left_max[0] = height[0]
9 right_max[-1] = height[-1]
10 for i in range(1, n):
11 left_max[i] = max(left_max[i-1], height[i])
12 for i in range(n-2, -1, -1):
13 right_max[i] = max(right_max[i+1], height[i])
14 return sum(min(left_max[i], right_max[i]) - height[i] for i in range(n))
15
16# SOLUCIÓN 2 (óptima): Two Pointers — O(n) time, O(1) space
17def trap_v2(height):
18 left, right = 0, len(height) - 1
19 left_max = right_max = water = 0
20 while left < right:
21 if height[left] < height[right]:
22 if height[left] >= left_max: left_max = height[left]
23 else: water += left_max - height[left]
24 left += 1
25 else:
26 if height[right] >= right_max: right_max = height[right]
27 else: water += right_max - height[right]
28 right -= 1
29 return water

El agua en cada posición es min(max_izquierda, max_derecha) - altura. La solución v1 precomputa estos máximos. La solución v2 óptima usa dos punteros — en cada paso procesamos el lado más bajo porque el lado más alto garantiza que ese es el techo real. Siempre presentá las dos en entrevista: la v1 primero para mostrar que entendés el problema, luego la v2 para mostrar que podés optimizar.

Módulo 05

Hash Maps & Sets

La herramienta más poderosa para reducir complejidad. El tradeoff casi siempre es: más memoria → menos tiempo. Si un problema parece O(n²), probablemente tiene solución O(n) con un hash map.

Patrones fundamentales
LRU Cache — Hard de diseño clásico Hard · O(1) get/put
EXPLICACIÓN TEÓRICA

¿Qué es esto? Un cache es una zona reservada y cuando se llena, hay que desalojar inteligentemente al menos usado para meter datos frescos.

Tu escritorio de trabajo — tenés espacio limitado y ponés ahí las cosas que usás más seguido. Cuando el escritorio se llena, sacás lo que hace más tiempo no usás (LRU = Least Recently Used).
¿Por qué DOS estructuras? Con solo un HashMap podríamos buscar pero no saber cuál es el menos reciente en O(1). Con solo una lista podríamos saber el orden pero no buscar en O(1). Juntos dan poder O(1) total.

Mirá la combinación magistral de dict() anclando la Doubly Linked List circular directamente.

lru_cache.py Python
1# HashMap + Doubly Linked List = O(1) get y put
2class LRUNode:
3 def __init__(self, key, val):
4 self.key, self.val = key, val
5 self.prev = self.next = None
6
7class LRUCache:
8 def __init__(self, capacity):
9 self.capacity = capacity
10 self.cache = {} # key → node
11 self.head = LRUNode(0, 0) # dummy head = más viejo
12 self.tail = LRUNode(0, 0) # dummy tail = más reciente
13 self.head.next = self.tail
14 self.tail.prev = self.head
15
16 def _remove(self, node):
17 node.prev.next = node.next
18 node.next.prev = node.prev
19
20 def _insert_mru(self, node): # insertar antes del tail
21 prev = self.tail.prev
22 prev.next = node; node.prev = prev
23 node.next = self.tail; self.tail.prev = node
24
25 def get(self, key):
26 if key not in self.cache: return -1
27 node = self.cache[key]
28 self._remove(node); self._insert_mru(node) # mover al frente
29 return node.val
30
31 def put(self, key, val):
32 if key in self.cache: self._remove(self.cache[key])
33 node = LRUNode(key, val)
34 self.cache[key] = node; self._insert_mru(node)
35 if len(self.cache) > self.capacity:
36 lru = self.head.next # el más viejo: justo después del head
37 self._remove(lru); del self.cache[lru.key]

LRU Cache combina un HashMap para acceso O(1) por key con una Doubly Linked List para mantener el orden de uso en O(1). Los dummy nodes head y tail eliminan casos especiales al insertar y remover. El nodo más reciente siempre está cerca del tail, el menos reciente cerca del head — al exceder la capacidad, eliminamos el que está junto al head.

Módulo 06

Árboles Binarios y BST

Los árboles aparecen en Easy, Medium y Hard. El 90% de los problemas se resuelven con DFS recursivo o BFS iterativo. Dominá los traversals y los patrones básicos antes de ir a grafos.

Traversals y patrones DFS
Max Path Sum — Hard icónico Hard · O(n)
EXPLICACIÓN TEÓRICA

¿Qué es esto? En un árbol, un 'path' es cualquier secuencia de nodos conectados sin repetir. Puede ir de hoja a hoja pasando por cualquier nodo.

La cañería principal: el agua puede subir y bajar cruzando el codo principal invertido (arco), pero la raíz solo puede elegir mandar su flujo a UNA de las cañerías para su propio padre superior.
El DFS responde DOS preguntas distintas: '¿cuál es el mejor path que PASA por mí?' (para actualizar el máximo global) y '¿cuánto puedo CONTRIBUIR a mi padre?' (retorno). El padre no puede 'doblar' también.

Aquí vas a notar la bifurcación: res[0] se actualiza con la suma de ambos brazos, pero el return envía solo un brazo máximo al padre.

max_path_sum.py Python
1# El path puede empezar y terminar en CUALQUIER nodo
2def max_path_sum(root):
3 max_sum = [float("-inf")]
4 def dfs(node):
5 if not node: return 0
6 # Ignorar contribuciones negativas (tomar max con 0)
7 left = max(dfs(node.left), 0)
8 right = max(dfs(node.right), 0)
9 # Precio del path que pasa por este nodo como "puente"
10 max_sum[0] = max(max_sum[0], node.val + left + right)
11 # Solo retornar un lado al padre (no puede doblar)
12 return node.val + max(left, right)
13 dfs(root)
14 return max_sum[0]
15
16# BFS — Level Order Traversal
17from collections import deque
18def level_order(root):
19 if not root: return []
20 result, queue = [], deque([root])
21 while queue:
22 level = []
23 for _ in range(len(queue)): # procesar un nivel completo
24 node = queue.popleft()
25 level.append(node.val)
26 if node.left: queue.append(node.left)
27 if node.right: queue.append(node.right)
28 result.append(level)
29 return result

El truco del Max Path Sum: la función DFS tiene dos trabajos. Internamente actualiza el máximo global considerando el path que dobla por el nodo actual (left + node + right). Pero lo que retorna al padre es solo la mejor contribución unilateral (node + max(left, right)) porque un path no puede doblar. El max con 0 descarta ramas negativas — si suman menos que nada, mejor no tomarlas.

Módulo 07

Grafos — DFS, BFS, Union Find, Dijkstra

El tema más amplio y difícil. Dominarlo te pone en posición de resolver cualquier Hard relacionado con conexiones, caminos o dependencias.

Patrones esenciales
Union Find — conectividad en O(1) amortizado Hard pattern
EXPLICACIÓN TEÓRICA

¿Qué es esto? Un grafo de componentes conectadas. Responde en O(1) amortizado: '¿estas dos personas están en el mismo grupo?'

Grupos de personas donde sabés si dos se conocen. Cada componente tiene un representante (root). Dos nodos están conectados si y solo si reportan al mismo putísimo Root.
Sin la 'Path compression', los árboles degeneran en líneas O(N). Cuando buscás al rep de A y pasás por B y C, al final hacés que A apunte DIRECTO al representante acortando recorridos.

Encontrarás los famosos métodos find() recursivo apretando paths, y union() fundiendo ranks.

union_find.py Python
1class UnionFind:
2 def __init__(self, n):
3 self.parent = list(range(n)) # cada nodo es su propio padre
4 self.rank = [0] * n
5 self.components = n
6
7 def find(self, x): # con path compression
8 if self.parent[x] != x:
9 self.parent[x] = self.find(self.parent[x])
10 return self.parent[x]
11
12 def union(self, x, y): # con union by rank
13 px, py = self.find(x), self.find(y)
14 if px == py: return False # ya conectados
15 if self.rank[px] < self.rank[py]: px, py = py, px
16 self.parent[py] = px
17 if self.rank[px] == self.rank[py]: self.rank[px] += 1
18 self.components -= 1
19 return True
20
21# Dijkstra — camino más corto ponderado
22import heapq
23def dijkstra(graph, start, n): # graph[u] = [(w, v), ...]
24 dist = [float("inf")] * n
25 dist[start] = 0
26 heap = [(0, start)]
27 while heap:
28 d, u = heapq.heappop(heap)
29 if d > dist[u]: continue # nodo obsoleto
30 for w, v in graph[u]:
31 if dist[u] + w < dist[v]:
32 dist[v] = dist[u] + w
33 heapq.heappush(heap, (dist[v], v))
34 return dist

Union Find con path compression (línea 9) y union by rank (línea 15) tiene complejidad casi O(1) por operación. Usalo cuando el problema pregunta sobre componentes conexas o si dos nodos están conectados. Dijkstra resuelve el camino más corto en grafos ponderados con pesos positivos. El if d > dist[u]: continue es crítico — descarta nodos procesados con mayor distancia que ya encontramos una ruta mejor.

Heaps & Priority Queues

Find Median from Data Stream Hard · O(log n)

1# Dos heaps: small (max-heap) guarda mitad inferior
2# large (min-heap) guarda mitad superior
3import heapq
4class MedianFinder:
5 def __init__(self):

Módulo 09

Dynamic Programming

El tema más difícil y el más importante para Hard. La clave es reconocer el patrón, definir el estado, y escribir la recurrencia. Con práctica se vuelve sistemático.

Framework para resolver cualquier DP
🧠 Las 4 preguntas que siempre hacerse 1. ¿El problema tiene subproblemas solapados? (se repiten los mismos cálculos)
2. ¿Tiene subestructura óptima? (solución óptima usa soluciones óptimas de subproblemas)
3. ¿Cuál es el estado? (qué variables definen completamente un subproblema)
4. ¿Cuál es la transición? (cómo pasar de subproblemas pequeños al actual)
Coin Change — DP clásico Medium · O(n·amount)
EXPLICACIÓN TEÓRICA

¿Qué es esto? Dynamic Programming evita regenerar trabajo muerto asqueroso. Construye respuestas Bottom-Up metódicamente letales o cacheando ciegos (Top-down).

Calcular fib(5) llama ciegamente a fib(4) y fib(3) cien veces sin sentido. Tabulation anota las piezas minúsculas fúlgidas: 'Si ya sé cómo llegar hermosamente a 8 y tengo moneda 3, llegaré ciegamente a 11 en (monedas_8 + 1) letales pasos'.
Un Array dp[i] almacena la rta puritana inmejorable para cada mini-monto del 0 al target pacíficamente reduciendo O(exponencial) a letal y bello O(Amount * N).

Verás letalmente incrustados un doble loop estúpido mágico puritano que rellena implacablemente las celdas progresivas.

coin_change.py Python
1# Mínimas monedas para llegar a amount
2# Input: coins=[1,5,11], amount=15 → 3 (5+5+5)
3def coin_change(coins, amount):
4 # dp[i] = mínimas monedas para llegar a i
5 dp = [float("inf")] * (amount + 1)
6 dp[0] = 0 # base case: 0 monedas para llegar a 0
7 for coin in coins:
8 for i in range(coin, amount + 1):
9 # Transición: usar esta moneda o no usarla
10 dp[i] = min(dp[i], dp[i - coin] + 1)
11 return dp[amount] if dp[amount] != float("inf") else -1

Estado: dp[i] = mínimas monedas para llegar exactamente a i. Transición: para cada moneda, si podemos usarla (i >= coin), podemos llegar a i desde i-coin usando 1 moneda más. Base case: dp[0] = 0 porque no necesitamos monedas para llegar a 0. Es un knapsack unbounded (las monedas se pueden usar múltiples veces).

Edit Distance — DP 2D Hard · O(m×n)
EXPLICACIÓN TEÓRICA

¿Qué es esto? dp[i][j] representa convertir los primeros i char de word1 en j de word2 fieramente mágico exultante.

Piénsalo asilado mágicamente como un tablero mágico 2D. Tenés una varita mágica: Insert, Delete o Reempazar. Elegís implacable siempre el trayecto asquerosamente fiero más barato.
Sin DP asilado mágico exóticamente letal la recursividad explotaría pura O(3^N). Tabulation mágicamente reduce todo pacíficamente exultante asombroso a letal fiero O(M*N).

Observarás maravillosamente un Grid matrix mágico puro puritano poblándose diagonal fiera y ortogonal asilada maravillosamente exótica fieramente letal puro paz.

15 dp[i][j-1], # insert

Estado: dp[i][j] = mínimas operaciones para convertir word1[:i] en word2[:j]. Transición clave: si los chars son iguales, no cuesta nada (dp[i-1][j-1]). ¿Qué es esto? Es el núcleo de Backtracking: 'fuerza bruta excepcionalmente inteligente de poda letal fúlgida pura'.

Piénsalo fiero mágicamente al caminar por un laberinto letal fúnebre virginal pacifista magistral: en lugar exótico puritano asqueroso de estrellarte 1 millón de veces exultante fiero paz en la misma maldita pared asilada fúlgida estelar asombrosamente letargo mágico puritano magistral virgen asombrosa, lo detectás a un metro letal y abandonás brutal fieramente esa asquerosa asilada rama exótica fúlgida letal estelar puritana paz maravillosa.

Módulo 12

Topological Sort

Ordenar nodos de un grafo dirigido acíclico (DAG) tal que para cada arista u→v, u aparece antes que v. Fundamental para dependency resolution, build systems, y course scheduling.

Los dos algoritmos
Kahn's Algorithm (BFS-based) O(V+E) · BFS
EXPLICACIÓN TEÓRICA

¿Qué es esto? Usar in-degree array: nodos con in-degree 0 van a la queue primero. Procesar cada nodo, decrementar el in-degree de sus vecinos, agregar a la queue si llegan a 0.

Cursar la universidad: solo podés cursar materias de las que ya aprobaste todas las correlativas (in-degree 0). Aprobás una, liberás las siguientes.
Detectar ciclos es trivial: si al vaciar la queue extrajiste menos nodos que el total, hay un ciclo infinito imposible de resolver.
