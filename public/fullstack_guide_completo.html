Módulo 01

JavaScript Fundamentos

El lenguaje que corre en el navegador y en Node. Sin dominar variables, tipos, funciones, asincronía y módulos, todo lo que construyas encima (React, Node) será frágil.

Variables y Scope
var vs let vs const JS
¿Qué es esto?

Antes de ES6 solo existía var. Las variables declaradas con var se "elevan" (hoisting) al inicio de la función, no del bloque. Eso permitía usar una variable antes de declararla y generaba bugs imposibles de rastrear.

Declarás var i en un loop, pero por el scope de función la misma i se reutiliza en callbacks. Los timeouts imprimen el valor final de i para todos. En producción: datos incorrectos, reportes de analytics equivocados.
let y const tienen scope de bloque. Cada iteración del loop tiene su propia variable. El callback captura el valor correcto. No más bugs de "la variable cambió antes de que se ejecutara".
Es como tener cajas numeradas (bloques). Con var había una sola caja compartida; con let cada habitación tiene su propia caja. No podés confundir qué guardaste dónde.
scope.js
1// var: scope de función — i se comparte
2for (var i = 0; i < 3; i++) {
3 setTimeout(() => console.log(i), 100); // 3, 3, 3
4}
5// let: scope de bloque — cada i es distinta
6for (let j = 0; j < 3; j++) {
7 setTimeout(() => console.log(j), 100); // 0, 1, 2
8}

• let y const son block-scoped. • const no reasigna la referencia, pero sí podés mutar objetos/arrays. • Usá const por defecto, let cuando necesites reasignar.

Temporal Dead Zone JS
¿Qué es esto?

Entre el inicio del bloque y la línea donde declarás let o const, la variable existe pero no podés accederla. Es la "zona muerta temporal". Si intentás leerla antes, JS lanza ReferenceError.

Con var obtenías undefined — silencioso, difícil de debuggear. Con let/const falla de inmediato.
El error te obliga a declarar antes de usar. No hay "undefined" silencioso. El código se rompe donde está el bug, no tres funciones más abajo.
Es como un contrato: la variable existe desde que entrás al bloque, pero no podés usarla hasta que la firmás (declarás). Si intentás usarla antes, el contrato no es válido.

• TDZ solo aplica a let y const. • var devuelve undefined por hoisting. • Siempre declarar en la parte superior del scope.

Tipos y Coerción
Tipos primitivos y de referencia JS
¿Qué es esto?

Los primitivos (number, string, boolean, null, undefined, symbol) se copian por valor. Los objetos y arrays se copian por referencia. [] === [] es false porque son dos referencias distintas en memoria.

Modificás un objeto que pasaste como prop a un componente hijo. El padre también cambia. React no detecta el cambio correctamente. Re-renders fantasmas o datos corruptos.
Entender que la comparación de objetos es por referencia. Para comparar contenido: spread, structuredClone, o librerías como lodash. Para inmutabilidad: no mutar, crear copias.
Los primitivos son fotocopias: cada una es independiente. Los objetos son direcciones de casas: si dos variables apuntan a la misma casa, cambiar el interior afecta a ambas.

• === compara referencias en objetos. • Usar spread o structuredClone para copias. • Inmutabilidad evita bugs en React.

Coerción implícita y truthy/falsy Importante
¿Qué es esto?

JavaScript convierte tipos automáticamente en contextos como if (x) o "5" + 3. Los valores "falsy" son: false, 0, "", null, undefined, NaN. Todo lo demás es truthy.

Un input vacío devuelve "". if (userInput) falla para string vacío pero 0 también es falsy — un contador en cero podría ocultar un formulario. El "billion dollar mistake" de JS.
Usar comparaciones explícitas: value === null, value === undefined, value === "". Para "tiene valor": value != null && value !== "". Evitar if (x) cuando 0 o "" son válidos.
Es como un guardia que interpreta: "no hay nadie" puede significar puerta cerrada, lista vacía o cero personas. Tenés que ser explícito sobre qué "nada" estás buscando.

• 0, "", null, undefined, NaN son falsy. • ?? (nullish coalescing) solo considera null/undefined. • Preferir === sobre ==.

null vs undefined vs NaN JS
¿Qué es esto?

Tres formas de "no hay nada" con significados distintos: undefined = no se asignó; null = asignado intencionalmente como vacío; NaN = resultado de una operación matemática inválida.

Mezclarlos genera lógica rota: typeof null es "object" (bug histórico). NaN === NaN es false. APIs que devuelven null vs undefined inconsistentemente.
Usar ?? para null/undefined. Para NaN: Number.isNaN(x). En APIs: elegir uno (null o undefined) y ser consistente. Optional chaining ?. evita crashes.
undefined = "nunca puse nada en la caja". null = "puse un cartel que dice vacío". NaN = "intenté calcular algo y el resultado no tiene sentido".

• ?? para default cuando null/undefined. • Number.isNaN() para detectar NaN. • ?. evita errores en cadenas.

Funciones
Declaración vs expresión vs arrow function JS
¿Qué es esto?

Función declarada: function foo() {} — hoisted, tiene su propio this. Función expresión: const foo = function() {} — no hoisted. Arrow: () => {} — no tiene this propio, hereda del scope.

En un callback de evento, this apunta al elemento DOM. Si usás function perdés el contexto de tu clase. Si usás arrow, this es el componente React. Confusión total en entrevistas.
Arrow functions heredan this del scope léxico. Perfectas para callbacks. Las declaradas tienen su propio this según quién las invoque. Elegir según el contexto.
La función declarada es un empleado que pregunta "¿quién me llamó?" cada vez. La arrow es uno que ya sabe "trabajo para este jefe" desde que nació.

• Arrow no tiene this, arguments, ni se puede usar como constructor. • Hoisting: declaradas sí, expresiones y arrow no. • Callbacks: preferir arrow.

Closures Importante
¿Qué es esto?

Una función que "recuerda" las variables del scope donde fue creada, incluso después de que ese scope ya no exista. Es el concepto más preguntado en entrevistas JS.

Sin entender closures: no entendés por qué un event handler sigue viendo un valor viejo. O por qué un factory function puede crear contadores independientes. Código que "debería funcionar" y no funciona.
La función mantiene una referencia al "entorno" (closure) donde nació. Cada llamada a createCounter() crea un nuevo closure con su propio count. Datos privados, factories, callbacks.
Es como una mochila que la función se lleva: guarda las variables del lugar donde nació. Donde sea que vaya, tiene acceso a lo que puso en la mochila.
closure.js
1function createCounter() {
2 let count = 0; // cada createCounter tiene su propio count
3 return () => ++count;
4}
5const c1 = createCounter();
6c1(); // 1
7c1(); // 2 — closure mantiene count

• Closure = función + referencias al scope externo. • Usado en: callbacks, factories, datos privados. • Cuidado con closures en loops (ya resuelto con let).

Callbacks y callback hell JS
¿Qué es esto?

Antes de Promises, la asincronía se manejaba con callbacks. Encadenar varias operaciones async generaba callbacks anidados (callback hell): código ilegible, difícil de mantener y de manejar errores.

Obtener usuario, luego sus pedidos, luego el detalle de cada pedido. Tres niveles de anidación. Errores que no se propagan. Código que se inclina hacia la derecha hasta desaparecer.
Promises y async/await. En lugar de callbacks anidados, encadenás .then() o usás await en secuencia. El código se lee como síncrono. Errores con try/catch.
Callbacks son como dar instrucciones por teléfono: "cuando termines, llamame y te digo el siguiente paso". Promises son como una lista de tareas: hacés una, marcás, pasás a la siguiente.

• Callback hell motivó Promises. • async/await es syntactic sugar sobre Promises. • Siempre manejar errores con try/catch o .catch().

Arrays y Objetos
map, filter, reduce JS
¿Qué es esto?

Métodos que transforman arrays sin mutar el original. map transforma cada elemento. filter filtra por condición. reduce reduce a un solo valor. Reemplazan la mayoría de los for loops imperativos.

For loops con push, índices que se desfasan al filtrar, acumuladores inicializados mal. Código verboso que oculta la intención. Bugs al mutar el array original sin querer.
Código declarativo: "quiero un array de nombres" → users.map(u => u.name). "Los activos" → users.filter(u => u.active). "La suma" → nums.reduce((a,b) => a+b, 0). Intención clara, sin mutación.
map = pasás cada item por una máquina y sale transformado. filter = un colador que deja pasar solo lo que cumple. reduce = una balanza que va sumando peso hasta dar el total.

• map retorna nuevo array. • reduce necesita valor inicial (segundo arg). • Preferir sobre for cuando la intención es transformar/filtrar/acumular.

Destructuring y spread/rest JS
¿Qué es esto?

Destructuring extrae propiedades o elementos en variables. Spread (...) expande. Rest recolecta el resto. Con valores por defecto: const { name = "Anónimo" } = user.

Acceder a propiedades anidadas con obj.a.b.c crashea si b es undefined. Copiar objetos con asignación solo copia la referencia. Mutación accidental en todo el código.
Destructuring con default: const { b = {} } = a. Spread para copiar: { ...obj } o [...arr]. Rest en parámetros: (a, ...rest) =>. Código más seguro y legible.
Destructuring es desarmar un paquete y poner cada cosa en su lugar. Spread es volver a empaquetar. Rest es "todo lo que sobra va en esta bolsa".

• ?? para default cuando null/undefined. • Spread en objetos: propiedades posteriores sobrescriben. • Rest debe ser el último parámetro.

Optional chaining (?.) y nullish coalescing (??) JS
¿Qué es esto?

?. cortocircuita si el valor es null/undefined: user?.address?.city no crashea si user o address son null. ?? usa el valor de la derecha solo cuando el de la izquierda es null o undefined.

Antes: user && user.address && user.address.city — verboso. O value || "default" que falla cuando value es 0 o "". Crash en producción al acceder a API que devolvió null.
Una línea: user?.address?.city ?? "Desconocido". El crash que eliminás: TypeError al leer propiedad de undefined. El default correcto: 0 y "" son valores válidos, ?? solo reemplaza null/undefined.
Optional chaining es como preguntar "¿existe la puerta? ¿existe la habitación?" antes de entrar. Nullish coalescing es "si no hay nada (null/undefined), usá esto otro".

• ?. retorna undefined si algo en la cadena es null/undefined. • ?? vs ||: ?? solo para null/undefined. • Usar en APIs y props opcionales.

Asincronía
Event Loop Importante
¿Qué es esto?

JavaScript es single-threaded. El Event Loop permite que operaciones asíncronas (setTimeout, fetch, I/O) no bloqueen el hilo principal. Las callbacks se encolan y se ejecutan cuando el call stack está vacío.

Sin entender el event loop: "¿por qué mi setTimeout(0) no corre ya?". O código síncrono pesado que congela la UI. O pensar que async = paralelo cuando en JS todo es un solo hilo.
Call stack → ejecuta código. Web APIs (o libuv en Node) manejan I/O. Callback queue recibe las callbacks. Event loop: cuando el stack está vacío, pasa una callback del queue al stack. Microtasks (Promises) tienen prioridad sobre macrotasks (setTimeout).
Un mozo (event loop) que atiende una mesa (call stack). Cuando la mesa está vacía, trae el siguiente pedido de la cocina (queue). Los pedidos urgentes (Promises) van primero.

• JS es single-threaded. • Microtasks antes que macrotasks. • setTimeout(fn, 0) no es inmediato, espera al siguiente tick.

Promises y async/await JS
¿Qué es esto?

Promise representa un valor futuro (pending → fulfilled o rejected). async/await permite escribir código asíncrono como si fuera síncrono. await pausa hasta que la Promise se resuelva.

Callbacks anidados, errores que no se capturan, código que salta de función en función. Difícil de leer y de debuggear.
async/await: código lineal, try/catch para errores. Detrás de escena sigue siendo Promises. La legibilidad mejora drásticamente. Es el estándar en código moderno.
Promise es un ticket de lavandería: "cuando esté listo te aviso". await es esperar en la puerta hasta que te llamen. async es "esta función va a tener que esperar en algún momento".
async.js
1async function fetchUser(id) {
2 try {
3 const res = await fetch(`/api/users/${id}`);
4 if (!res.ok) throw new Error('Fail');
5 return await res.json();
6 } catch (e) { console.error(e); }
7}

• async siempre retorna Promise. • await solo dentro de async. • fetch no rechaza en 4xx/5xx — chequear res.ok.

Promise.all vs race vs allSettled JS
¿Qué es esto?

Promise.all: todas en paralelo, falla si una falla. Promise.race: la primera que termina (éxito o error). Promise.allSettled: todas terminan, devuelve estado de cada una.

Hacer 5 fetches en secuencia con await: lento. Hacerlos en paralelo sin control: una falla y perdés todo. Necesitás saber cuáles fallaron sin abortar el resto.
all: cuando necesitás todas (ej. datos para un dashboard). race: timeouts, "la respuesta más rápida". allSettled: cuando querés resultados parciales (ej. múltiples APIs, mostrar lo que sí llegó).
all = esperar a que todos los invitados lleguen a la fiesta. race = el primero que toca la puerta gana. allSettled = anotar quién llegó y quién no, sin cancelar la fiesta.

• all: rechaza si alguna rechaza. • allSettled: siempre resuelve con array de {status, value/reason}. • race: útil para timeouts.

Módulos
CommonJS vs ES Modules JS
¿Qué es esto?

CommonJS: require() y module.exports — síncrono, usado en Node clásico. ES Modules: import y export — estándar, estático, usado en navegadores y Node moderno.

Conviven en el ecosistema. Un paquete en npm usa require, tu proyecto usa import. Errores de "Cannot use import outside a module". Configuración de package.json type: "module".
En proyectos nuevos: ES Modules. En Node: "type": "module" en package.json o extensión .mjs. Named exports permiten tree-shaking. Default export para un solo valor principal.
CommonJS es como ir al almacén cada vez que necesitás algo. ES Modules es como tener la lista de compras antes de salir: el bundler sabe exactamente qué necesita.

• import es estático (parseado al cargar). • require es dinámico. • Named exports mejor para tree-shaking.

Named exports vs default exports JS
¿Qué es esto?

Named: export const foo — importas con { foo }. Default: export default Foo — importas con cualquier nombre. Un módulo puede tener varios named y un solo default.

Default export permite renombrar al importar: confusión en equipos. "¿Cómo se llama el componente que exporta este archivo?". Refactors que rompen imports en toda la app.
Named exports: el nombre es explícito, el IDE autocompleta, mejor para tree-shaking. Default para el "export principal" de un archivo (ej. un componente React). Convención: en React, default para componentes de página.
Named es como un menú con platos nombrados. Default es el "plato del día" — puede ser lo que sea, pero el restaurante solo tiene uno por día.

• Named: más explícito, mejor DX. • Default: útil para componentes/una sola cosa. • Consistencia en el equipo importa más que la elección.

Módulo 02

TypeScript Esencial

JavaScript con tipos. Detecta errores antes de que lleguen a producción. El estándar en proyectos React y Node serios.

Por qué TypeScript
El costo real de errores de tipo TS
¿Qué es esto?

Un bug de tipo en producción: función que esperaba string recibe number. Resultado: NaN, undefined, crash. En una app de pagos: montos incorrectos. En una de citas: fechas rotas. El fix cuesta 10x más que prevenirlo.

Sin tipos, el error aparece cuando el usuario hace la acción específica. QA no lo encuentra. El cliente reporta. Hotfix urgente. Pérdida de confianza.
TypeScript marca el error en el IDE, en el momento de escribir. El compilador rechaza el build. No llega a producción. Inversión en tipos = menos bugs = menos costos.
Es como revisar el contrato antes de firmar vs descubrir la cláusula mala cuando te demandan. TypeScript es el abogado que lee todo antes.

• Errores de tipo en runtime cuestan 10-100x más. • TS compila a JS — no es magia, es una capa. • Adopción gradual: renombrar .js a .ts.

Cómo TypeScript se compila a JS TS
¿Qué es esto?

TypeScript es un superset: todo JS válido es TS válido. El compilador (tsc) borra los tipos y emite JS. Los tipos no existen en runtime. Es una capa de verificación estática.

Pensar que los tipos "protegen" en runtime. No. Si el dato viene de una API sin validar, TS no puede ayudarte en ejecución.
Compilación elimina tipos. Para runtime: validar con Zod o similar. TS + Zod = tipos en compile-time y validación en runtime.
TS es el inspector de construcción que revisa los planos antes de construir. Una vez construida la casa (JS), él ya no está. Los planos no son la casa.

• tsc compila .ts → .js. • Tipos desaparecen en el output. • Validar datos externos en runtime.

Tipos Básicos
any vs unknown vs never Importante
¿Qué es esto?

any desactiva el checker — cualquier cosa pasa. unknown es "no sé qué es, tengo que verificar". never es "nunca ocurre" (funciones que no retornan, exhaustividad).

any es trampa: te deja compilar código que crashea. Un any en una función propaga y contagia. Perdés todos los beneficios de TS.
unknown cuando el tipo viene de fuera: API, DOM. Hacés type narrowing antes de usar. never para exhaustividad en switch. Evitar any: usar unknown o genéricos.
any es "confío en vos" (peligroso). unknown es "mostrame el DNI antes". never es "esta puerta no lleva a ningún lado".

• any = escape hatch, usar con cuidado. • unknown + type guard antes de usar. • never en exhaustividad.

Interfaces y Types
interface vs type TS
¿Qué es esto?

Ambos definen formas de objetos. interface puede extenderse con declaration merging. type puede ser unions, primitivos, tuples. En la práctica: interface para objetos, type para unions y utilidades.

Mezclarlos sin criterio. type para todo cuando interface es más legible para objetos. interface para unions (no se puede).
interface para contratos de objetos (props, API responses). type para unions (|), intersections (&), y utilidades (Pick, Omit). La respuesta definitiva: preferir interface para objetos, type para unions.
interface es como un formulario de contrato estándar. type es como un contrato a medida que puede combinar varios formularios.

• interface: declaration merging, extends. • type: unions, primitivos, tuples. • Readonly y Optional con ?.

Generics
El problema sin generics TS
¿Qué es esto?

Sin generics: una función que retorna array pierde el tipo del elemento. function first(arr: any[]): any — el tipo se pierde. O duplicás la función para cada tipo.

Código duplicado o tipo perdido. const x = first(users) → x es any. No autocompletado. No type safety.
Generics: function first<T>(arr: T[]): T. El tipo se infiere. first(users) retorna User. El patrón más preguntado en entrevistas TS.
Generic es como un molde: "voy a darle forma a lo que me pases, pero no sé qué material hasta que me lo des".

• <T> es el parámetro de tipo. • Constraints con extends. • Inferencia automática cuando es posible.

Utility Types
Partial, Pick, Omit, Record TS
¿Qué es esto?

Tipos que transforman otros tipos. Partial hace todo opcional. Pick selecciona campos. Omit elimina campos. Record crea objeto con keys y valores tipados.

Crear manualmente tipos para "actualización parcial" o "solo estos campos". Duplicación de tipos. Mantenimiento pesado.
Partial<User> para updates. Pick<User, 'id'|'name'> para proyecciones. Omit<User, 'password'> para excluir. Record<string, number> para mapas. Los más usados en proyectos reales.
Son como herramientas de carpintería: no construís una nueva pieza desde cero, transformás la que tenés.

• Partial, Required, Readonly

• ReturnType, Parameters para funciones.

Módulo 03

HTML & CSS Producción

El fundamento de toda web. Semántica, layout, responsive. Sin esto, React es solo divs con esteroides.

HTML Semántico
div-itis y sus consecuencias HTML
¿Qué es esto?

Todo en divs: sin estructura. Los lectores de pantalla no entienden la jerarquía. Google no sabe qué es importante. Mantener código de divs anidados es un infierno.

SEO pobre, accesibilidad rota, mantenibilidad cero. Cambiar un layout es adivinar qué div hace qué.
header, main, section, article, nav, aside, footer. Cada uno tiene significado. Los lectores de pantalla navegan por landmarks. Google entiende la estructura.
div-itis es como escribir un libro sin capítulos ni párrafos: todo en un bloque. Los elementos semánticos son la estructura de un libro real.

• main para contenido principal. • nav para navegación. • label asociado a inputs.

Box Model y Flexbox
box-sizing: border-box Importante
¿Qué es esto?

Por defecto: width + padding + border = tamaño total. Con border-box: width incluye padding y border. El ancho que ponés es el ancho real.

width: 100% + padding = overflow. Layouts que se rompen. Cálculos manuales para compensar.
* { box-sizing: border-box; } al inicio de toda hoja de CSS. O en el reset. Por qué toda hoja empieza con esto.
content-box: comprás una caja de 100cm y le agregás relleno — la caja crece. border-box: la caja es 100cm total, el relleno va adentro.

• Content, padding, border, margin. • border-box = width incluye todo. • flex-grow, flex-shrink, flex-basis.

CSS Grid y Responsive
Cuándo Grid, cuándo Flexbox CSS
¿Qué es esto?

Flexbox: una dimensión (fila o columna). Grid: dos dimensiones (filas y columnas). La regla: Flexbox para componentes (header, card), Grid para layouts de página.

Usar Flexbox para layouts 2D: hacks con flex-wrap. Usar Grid para un botón con icono: overkill.
Flexbox: navegación, cards, centrado. Grid: layout principal, galerías, dashboards. auto-fill/auto-fit para responsive sin media queries.
Flexbox es una fila de sillas. Grid es un cuaderno de cuadrícula. ¿Organizás una fila o una tabla?

• Mobile-first: base móvil, media queries para desktop. • rem vs em. • Variables CSS para theming.

Módulo 04

React Core

Componentes, estado, efectos, hooks. La base de la UI moderna. Sin entender el modelo mental de React, todo es magia negra.

Fundamentos
JSX y componentes funcionales React
¿Qué es esto?

JSX es syntactic sugar para React.createElement. Los componentes funcionales ganaron sobre las clases: menos boilerplate, hooks, mejor optimización.

Pensar que JSX es HTML. No: es JavaScript. className, no class. Las props son inmutables.
Componente = función que retorna JSX. Props = parámetros. Flujo unidireccional: datos bajan, eventos suben. Prop drilling cuando hay muchos niveles.
JSX es como escribir HTML pero con superpoderes: podés meter variables, loops, condicionales. Todo se compila a llamadas a funciones.

• Props inmutables. • Componentes = funciones. • key en listas para evitar bugs.

Estado y Efectos
useState y useEffect Importante
¿Qué es esto?

useState: estado que causa re-render al cambiar. No mutar directamente. useEffect: side effects (fetch, suscripciones). Dependencias: el array de deps controla cuándo corre.

Mutación directa: setUser(user.name = 'x'). No re-render. useEffect sin deps: loop infinito. Deps incorrectas: closures stale.
Siempre setState con nuevo valor. useEffect: dependencias exactas. Cleanup en return para suscripciones. useRef para valores que no causan re-render.
useState es el cuaderno del componente: cuando escribís algo nuevo, el componente se redibuja. useEffect es "cuando X cambie, hacé Y".

• No mutar estado. • Reglas de Hooks: orden constante, no en condicionales. • useRef para DOM o valores persistentes.

Context y Patrones
useContext y Custom Hooks React
¿Qué es esto?

Context: estado global sin librería. Custom Hooks: extraer lógica reutilizable (useState + useEffect en una función).

Prop drilling: pasar props por 5 niveles. Context para todo: re-renders innecesarios.
Context para tema, auth, idioma. Custom hooks para lógica de fetch, formularios, timers. Compound components para componentes flexibles.
Context es como un altavoz: cualquiera en la sala escucha. Custom Hook es una receta que guardás para reutilizar.

• Context + Provider. • Custom Hooks: use + nombre. • React Hook Form + Zod para formularios.

Módulo 05

Node.js & Express

El backend en JavaScript. Event loop, non-blocking I/O. Express: routing, middlewares, REST API. El stack que usa el 80% de las startups.

Node.js Core
Event loop y non-blocking I/O Node
¿Qué es esto?

Node es single-threaded. El event loop permite manejar miles de conexiones: cuando una operación I/O espera, el hilo atiende otras. No threading tradicional.

Código síncrono bloquea todo. Un loop pesado congela el servidor. No hay paralelismo real para CPU.
I/O no bloqueante: callbacks, Promises. Para CPU intensiva: worker threads o separar servicios. Streams para archivos grandes.
Un mozo que atiende muchas mesas: mientras espera un pedido de la cocina, sirve a otras. No hay un mozo por mesa.

• require, module.exports. • fs, path, os. • Streams para archivos grandes.

Express
Middlewares y request/response cycle Express
¿Qué es esto?

Request llega → middlewares en orden (auth, body parser, router) → response. Middleware: función que recibe req, res, next. next() pasa al siguiente.

Orden incorrecto: body parser después del router. Auth sin next(): la request se queda colgada. Error handler con 4 params (err, req, res, next).
Orden: CORS, body parser, router, error handler al final. Middleware de auth: verificar token, next() o 401. Error handler: middleware de 4 parámetros.
Middlewares son como filtros de agua: cada uno hace algo (quitar cloro, quitar sedimentos) y pasa al siguiente. El orden importa.

• app.use para middleware global. • router.get/post/put/delete. • params, query, body.

REST API y Auth
JWT y convenciones REST API
¿Qué es esto?

REST: recursos como sustantivos (GET /users), verbos HTTP. JWT: token firmado con payload (userId, exp). Access token corto, refresh token largo. bcrypt para passwords.

Passwords en texto plano. Tokens sin expiración. Endpoints inconsistentes (/users vs /user).
bcrypt.hash para guardar. JWT con exp. Refresh token en httpOnly cookie. Status codes correctos: 200, 201, 400, 401, 404, 500.
JWT es como un pase de evento: tiene tu nombre, fecha de expiración, y una firma que no se puede falsificar.

Módulo 06

SQL & PostgreSQL

Datos relacionales. Tablas, JOINs, agregaciones, transacciones. La base de la mayoría de las apps.

Fundamentos y JOINs
INNER JOIN vs LEFT JOIN SQL
¿Qué es esto?

INNER: solo filas que coinciden en ambas tablas. LEFT: todas las de la izquierda, null si no hay coincidencia. El corazón del modelo relacional.

Usar INNER cuando necesitás LEFT: perdés registros. El error N+1: un query por cada usuario para obtener sus pedidos.
LEFT cuando el "padre" puede no tener "hijos". JOIN con agregación o subquery para evitar N+1. Índices en foreign keys.
INNER es como una fiesta: solo entran los que tienen pareja. LEFT es como una lista: todos los de la izquierda, con o sin pareja.

• GROUP BY: agrupar por criterio. • HAVING: filtrar grupos. • Índices en columnas de búsqueda.

Transacciones
ACID y transacciones SQL
¿Qué es esto?

ACID: Atomicidad (todo o nada), Consistencia, Aislamiento, Durabilidad. BEGIN, COMMIT, ROLLBACK. En fintech: transferencia = debitar + acreditar en una transacción.

Operaciones sin transacción: si falla la segunda, la primera queda aplicada. Dinero que desaparece o se duplica.
BEGIN para iniciar. Todas las operaciones. COMMIT si todo bien. ROLLBACK si algo falla. Evitar deadlocks: orden consistente de locks.
Transacción es como un contrato: o se firman todas las páginas o ninguna. No hay medio contrato.

• BEGIN, COMMIT, ROLLBACK. • Deadlocks: mismo orden de locks. • Pool de conexiones.

Módulo 07

APIs REST & Data Fetching

HTTP, fetch, axios, TanStack Query. Cómo consumir APIs desde el frontend. CORS, cache, estados de carga.

HTTP y Fetch
fetch vs axios y CORS HTTP
¿Qué es esto?

fetch: nativo, no rechaza en 4xx/5xx. axios: rechaza en error, interceptors. CORS: el navegador bloquea requests cross-origin salvo que el servidor envíe headers correctos.

fetch: if (!res.ok) throw. CORS en desarrollo: el backend no envía Access-Control-Allow-Origin. "Blocked by CORS" sin entender por qué.
Backend: app.use(cors()). fetch: siempre chequear res.ok. Axios: interceptors para token. AbortController para cancelar.
CORS es como el guardia del club: el navegador pregunta "¿puedo entrar?" y el servidor responde con los headers. Sin permiso, no pasa.

• Los 3 estados: loading, error, data. • fetch no rechaza en 4xx. • Content-Type: application/json.

TanStack Query
useQuery y useMutation Importante
¿Qué es esto?

TanStack Query: cache automático, refetch, estados (loading, error, data). useQuery para GET. useMutation para POST/PUT/DELETE con invalidación de cache.

fetch manual: loading state, error state, cache. Código repetido en cada componente. No hay sincronización entre ventanas.
useQuery: key, fetcher. Cache por key. useMutation: onSuccess, invalidateQueries. Optimistic updates para UX fluida.
Es como un bibliotecario: pedís un libro (query), él lo busca, lo cachea, y si alguien más lo pide lo tiene listo. Si lo actualizás (mutation), invalida el cache.

• queryKey para cache. • invalidateQueries después de mutation. • localStorage vs httpOnly para tokens.

Módulo 08

Testing

Jest, React Testing Library, Supertest. Tests que no se rompen con cada refactor. Pirámide de testing.

Filosofía y Jest
Testear comportamiento, no implementación JS
¿Qué es esto?

Testing Library: tests desde la perspectiva del usuario. Qué ve, qué hace. No: "¿llamó a setState?". Sí: "¿muestra el mensaje después de hacer click?".

Tests que buscan clases internas, implementación. Refactor rompe tests. Tests frágiles que no aportan confianza.
getByRole, getByLabelText. userEvent sobre fireEvent. Mock de fetch/axios. No testear detalles de implementación.
Testear implementación es como revisar si el chef usó exactamente 3 cucharadas. Testear comportamiento es probar si la comida está bien.

• describe, it, expect. • jest.fn(), jest.mock(). • beforeEach para setup.

Módulo 09

Docker & Deploy

Contenedores, Dockerfile, docker-compose. Deploy en Railway, Render, Vercel. CI/CD con GitHub Actions.

Docker y Deploy
Imagen vs Contenedor y Dockerfile Docker
¿Qué es esto?

Imagen = plantilla. Contenedor = instancia en ejecución. Dockerfile: FROM, WORKDIR, COPY, RUN, CMD. .dockerignore para no copiar node_modules.

"En mi máquina funciona". Dependencias distintas, Node version distinta. Deploy que falla.
Docker: mismo ambiente en todos lados. Multi-stage: build en una imagen, run en otra más ligera. Layer caching: orden de instrucciones para velocidad.
Imagen es la receta. Contenedor es el plato servido. Dockerfile es la receta escrita. Todos cocinan lo mismo.

• docker-compose para app + DB. • Volúmenes para persistencia. • Railway, Render, Vercel.
